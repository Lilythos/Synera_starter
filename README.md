## 算法说明

### BFS 寻路

`Pathfinder::findPath(...)` 使用 BFS 在 8x8 网格上搜索路径。搜索时会跳过：

- 棋盘外位置。
- 已访问位置。
- 被其他单位占用的位置。

移动时只走路径中的下一格，因此每次 tick 最多移动一格。

战斗移动时，单位会优先尝试走向目标敌人上下左右的相邻空格；如果这些格子已经被其他单位占满，则会在棋盘上寻找一个“可达、未被占用、并且比当前位置更靠近目标”的空格继续移动。这样可以避免近战单位把目标围住后，后排单位完全停住不动，同时仍然满足不能穿过单位和不能重叠的规则。

### 索敌

`Targeting::selectTarget(...)` 遍历棋盘上的敌方存活单位，按距离和优先级选择目标。距离使用欧氏距离平方，避免浮点开方。

### 自动战斗 tick

`CombatSystem::updateBoard(...)` 会遍历棋盘单位，每个单位执行：

1. 推进攻击和移动冷却。
2. 若 Mana 满，则进入 `Casting` 并释放主动技能。
3. 若目标在攻击范围内，按攻击间隔普攻。
4. 若目标不在攻击范围内，按移动间隔向目标附近空格移动；若目标周围被堵住，则移动到更靠近目标的可达空格。
5. 单位死亡后从棋盘移除。

### 羁绊计算

`SynergySystem` 统计我方场上存活单位的 `traits`。达到阈值后，为对应单位添加临时加成。刷新时先清空旧加成，再重新计算，避免重复叠加。

### 存档读档

存档使用 Qt JSON：

- `QJsonObject`
- `QJsonArray`
- `QJsonDocument`

读档时先创建所有单位，并建立“存档 id -> 新 Unit*”映射；随后再恢复棋盘、Bench 和商店位置，避免新对象 id 与旧 id 不一致造成恢复失败。

## AI 使用说明

本项目开发过程中使用了 AI 辅助，但需求拆分、规则取舍、体验调整和最终验收判断均由我完成。我没有把 AI 输出作为不可解释的黑盒使用，而是逐步阅读、编译、运行和检查每个模块，确认其与 PA 要求和实际玩法一致。

具体使用方式：

- 我先将 PA 要求交给 AI，请 AI 生成阶段一至阶段三的开发顺序。
- 我根据 PA 文档检查 AI 给出的计划，决定哪些规则应先实现、哪些设计需要调整。
- 每个功能完成后，我会通过编译、测试或实际运行游戏检查效果，再把发现的问题反馈给 AI 辅助定位和修改。
- 英雄设计、UI 紫金风格、纯文字英雄展示、游玩攻略、准备阶段结算后回到我方最底下一排等体验设计，是我在反复游玩后提出的设计建议的。
- Bench 可以从棋盘拖回空位、装备拖拽穿戴、商店英雄池调整、敌人成长速度调整、失败金币奖励调整等，也是我根据实际游玩体验提出的更有利于玩家交互的改动。
- 实现不理想或有 bug 的地方，例如我方英雄移动到敌方区域后无法拖动、升星后装备消失、Qt lambda 连接崩溃、测试预期与规则不一致等，均由我先发现问题，再要求 AI 辅助修复。

我重点掌握并检查的 AI 辅助核心模块：

1. `CombatSystem`

`CombatSystem` 是阶段二自动战斗的核心模块，负责把索敌、攻击、回蓝、施法、移动和死亡移除串成一个 tick 流程。它本身不保存玩家金币、商店、GUI 等全局状态，只接收 `Board` 和 `Unit`，直接操作棋盘上的单位，因此职责比较集中。

它的主要运行流程是：

- `GameWindow` 用 `QTimer` 每隔一段时间调用 `Game::updateCombat(deltaMs)`。
- `Game::updateCombat()` 再调用 `CombatSystem::updateBoard(board, deltaMs)`。
- `updateBoard()` 遍历当前棋盘上的单位，跳过空指针、死亡单位和已经不在棋盘上的单位。
- 对每个单位调用 `updateUnit()`。
- `updateUnit()` 先调用 `advanceCombatTimers(deltaMs)` 推进攻击和移动冷却。
- 如果单位 `mana >= maxMana`，就把状态设为 `Casting`，调用单位自己的 `castSkill(...)`。这里体现多态：枪手、治疗师、战神的满蓝技能由各自英雄类实现。技能释放成功后调用 `resetMana()` 清空法力。
- 如果没有释放技能，就调用 `Targeting::selectTarget(...)` 找最近敌人。
- 如果目标在攻击范围内，并且攻击冷却已好，就执行 `basicAttack()`：设置攻击状态、计算伤害、扣目标 HP、给攻击者回蓝，并记录动作。
- 如果目标不在攻击范围内，并且移动冷却已好，就通过 `Pathfinder` 找到目标附近可达空格，然后单步移动，避免穿过单位和重叠。
- 如果攻击导致目标死亡，死亡单位会从 `Board` 移除，避免后续继续被寻路或索敌使用。

我理解这个模块的关键点是：它把“单位下一步该做什么”拆成固定优先级，顺序是满蓝技能优先，其次攻击，最后移动。这样符合自走棋逻辑，因为技能释放不应该被普通攻击或移动延后。GUI 不参与战斗判断，只负责显示 `Unit` 的位置、血条、蓝条和状态。

我检查过的边界问题包括：死亡单位是否会继续留在棋盘上、单位是否可能走到被占用格子、满蓝后是否清空 Mana、攻击和移动是否受冷却控制。

2. `SynergySystem`

`SynergySystem` 是阶段三羁绊系统的核心模块，负责统计我方场上单位的 `traits`，并根据人数阈值给单位添加临时加成。它只统计玩家场上存活单位，不统计敌人、商店和备战区，因此符合 PA 中“羁绊由我方上场单位触发”的逻辑。

它的主要运行流程是：

- 先遍历 `Board` 上的单位，筛掉空指针、死亡单位和敌方单位。
- 对每个我方存活单位读取 `traits()`，统计每个 trait 出现的人数。
- 在应用新羁绊前，先对所有我方场上单位调用 `resetSynergyBonuses()`，清空上一轮羁绊加成。
- 根据统计结果判断是否激活羁绊：
  - 枪手 2/4：所有我方单位 ATK 增加 10/20。
  - 治疗师 2：战斗开始前所有我方单位恢复 25 HP。
  - 殉道者 2：殉道者死亡治疗额外增加。
  - 模仿者 2：模仿者获得攻击力加成，并配合自身动作模仿机制。
  - 战意 3：所有我方单位攻击间隔减少 100ms。
- `activeSynergyDescriptions(...)` 使用同一套统计逻辑生成 GUI 展示文字，让界面显示和实际加成保持一致。

我理解这个模块的关键点是：羁绊加成不是永久改基础属性，而是临时加成。每次刷新时先清空旧加成，再按当前阵容重新计算，这样可以避免玩家频繁拖动单位、升星、读档后导致羁绊重复叠加。比如一个单位本来因为枪手羁绊 ATK +10，如果下次刷新不先清空，再加一次就会变成 +20，这是错误的。

我检查过的边界问题包括：是否错误统计敌方单位、是否错误统计 Bench 单位、羁绊描述和实际加成是否一致、刷新多次是否会重复叠加。
